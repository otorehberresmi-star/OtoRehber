# Supabase SQL Source Of Truth

`complete_schema_migration.sql` is the canonical migration for the app schema.
Use it as the first reference when setting up or repairing the database.

## Files

- `complete_schema_migration.sql`: Main schema/RLS migration for OtoRehber.
- `vehicle_specs_import.generated.sql`: Generated import for technical vehicle specs.
- `vehicle_specs_missing_catalog.generated.sql`: Generated catalog backfill for models referenced by specs.
- `vehicle_variants_catalog_import.generated.sql`: Generated multi-stage brand/model/engine/trim catalog import from the user-provided JSON dataset.
- `daily_comparison_specs_patch.sql`: Limits the daily pair to models with technical specs and refreshes today's cached pair.
- `security_preferences_schema.sql`: Adds the persistent profile privacy preference and masks private profiles in the public view.
- `post_topic_tags_schema.sql`: Persists the topic types selected for posts.
- `helpful_vote_notifications.sql`: Shows who gave helpful votes and removes the notification when a vote is withdrawn.
- `review_downvotes_schema.sql`: Adds downvotes to vehicle reviews.
- `user_follows_schema.sql`: Adds persistent user following relationships and RLS.
- `follow_notifications.sql`: Notifies users when someone follows them.
- `legal_consents_schema.sql`: Stores immutable, versioned registration acknowledgements for terms, KVKK and privacy documents.
- `cron_push_setup.sql`: Deployment-specific cron setup. Run separately after replacing project-specific placeholders.
- `cron_email_setup.sql`: Deployment-specific email notification cron setup for Resend. Run separately after replacing project-specific placeholders.
- `cleanup_mock_data.sql`: Destructive cleanup helper. Do not run as part of normal setup.

The smaller `*_schema.sql` files can stay in the repository as historical/module-level fragments, but production setup should not depend on running them one by one.

## Change Rule

When the app needs a schema, RLS, trigger, function, storage, or notification-table change:

1. Update `complete_schema_migration.sql` first.
2. Update a smaller `*_schema.sql` fragment only if it is still useful as module-level documentation.
3. Run the canonical migration in Supabase SQL Editor.
4. Keep deployment-specific setup, such as cron secrets, outside the canonical migration.

## Edge Functions

- `functions/dispatch-push-notifications`: due notification jobs and Expo push dispatch.
- `functions/dispatch-email-notifications`: due notification jobs and Resend email dispatch.
- `functions/otorehber-api`: mobile-facing secure API gateway for operations that
  should use server-side secrets instead of direct client table/storage access.

## Notification Deployment Checklist

Native push uses Expo push tokens and the Expo Push API because the mobile app is
an Expo/EAS app. This remains necessary in production unless the app is migrated
to direct APNs/FCM device-token handling.

Weekly digest, campaign, and vehicle reminder notifications are generated by
`generate_due_notification_jobs()`. That database function is not a scheduler by
itself. Without the Supabase cron setup, these jobs are generated only by
foreground app triggers such as opening the notifications feed.

Before store release:

1. Run `complete_schema_migration.sql` in Supabase SQL Editor.
2. Deploy `otorehber-api` with JWT verification, and deploy `dispatch-push-notifications` plus `dispatch-email-notifications` with `--no-verify-jwt` because they are protected by cron-only dispatch secrets.
3. Set `DISPATCH_PUSH_SECRET`, `DISPATCH_EMAIL_SECRET`, `RESEND_API_KEY`, `EMAIL_FROM`, and optionally `EMAIL_REPLY_TO` as Supabase secrets.
4. Replace placeholders in `cron_push_setup.sql` and `cron_email_setup.sql`, then run both files in Supabase SQL Editor. These scheduled jobs are what make weekly digest and campaign notifications run in the background.
5. Verify on a real iOS/Android device that Expo tokens are written to `push_tokens` and that a queued notification reaches the phone.
6. Verify with a test account that `email_enabled=true` sends through Resend and writes `email_sent_at`.

SMS is intentionally not part of the notification stack because OtoRehber does
not collect or use phone numbers for messaging.
